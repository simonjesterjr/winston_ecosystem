# Data Storage & Reconciliation (Parquet Winston Standard + PG Metadata Only)

This is a core, non-negotiable part of the current design (from user feedback on the initial plan + approved revision).

## Primary Artifact: Parquet Files ("Winston EOD Standard")
- Location (in DM): `data_manager/data/markets/{SYMBOL}/bars.parquet` (or lightly partitioned subdirs for very large histories).
- Contents per file (sorted by date):
  - Raw: date, open, high, low, close, volume (adjusted where the source provides it).
  - Derived (calculated by DM, not consumers): atr_17 (simple, matching WUT), plus the exact SMA/EMA/WMA periods currently supported by WUT strategies and expected for Wv2 evaluation (exact list to be pulled from WUT `strategies/` + `indicator_calculator` during implementation).
- This is the portable, canonical form that travels between monoliths (file mount in podman, HTTP download, or future object storage).
- DM is the writer / owner of freshness and correctness of both raw history (3y+ + latest) and the baked-in derivatives.

## PostgreSQL Role (Metadata Only)
- `Market` (symbol, name, trading_market, tick config, etc. — reuse WUT shape).
- `DataCoverage` (per market: earliest/latest dates, bar_count, indicators_present as jsonb, last_reconciled_at, source="eodhd").
- `DownloadRun`, `DownloadTask` (orchestration, step status, errors, timings — for Cromwell visibility and human ops).
- `Consumer` registry (WUT, future Wv2: base_url, api_token, last_verified).
- Config, audit, whatever the monolith needs to run.
- **No** heavy OHLCV or indicator time-series rows in PG tables. Those live in the parquet files.

## Reconciliation (The "Hidden" but Critical Feature)
- `DataReconciliationService` (or equivalent) walks the `data/markets/*` parquet tree (or a manifest/sidecars).
- For each symbol it extracts actual min/max date + present derived columns (via DuckDB or lightweight metadata).
- It then ensures `Market` exists and a `DataCoverage` row accurately reflects reality (upsert).
- When/why it runs:
  - On DM app boot (or a cheap guard job).
  - After every successful download/standardize run for a market.
  - Explicitly via rake/task (`data:reconcile` or `data:reconcile[SYMBOL]`).
- Why it matters:
  - "git clone the DM repo + drop your real (or sample) parquet data files in the data/ tree" → run reconcile → PG now knows the data exists; DM won't think everything is missing and re-download the world.
  - "Someone updated an old data file in place (added newer bars)" → reconcile detects the new latest date without treating it as a brand-new data set.
  - Shipping the repo with a few small reference/sample parquets for common symbols (or instructions + a generator) becomes viable.
  - Makes the "data folder as the source of truth for what data we actually have" a reality.

## Write / Update Flow (High Level)
1. Need more data for SYMBOL (gap or 3y policy or latest bar).
2. Fetch exact range from EODHD (single source in v1).
3. (Re)standardize: combine with any existing local data for the symbol if present, compute/recompute the required trailing derivatives (ATR + MAs) so they are correct across the whole series.
4. Write the Winston-standard parquet atomically (temp + rename, or DuckDB features).
5. Update PG coverage.
6. (Re)run reconciliation for good measure.
7. Notify Cromwell + make available (parquet file is now the artifact).

## Git / Repo Considerations
- Parquet files are binary and can be large → `.gitignore` the bulk of `data_manager/data/` (with a README or .gitkeep + notes in ecosystem/deployment/ explaining how to obtain or generate samples).
- Small reference parquets for 1–3 symbols (enough for tests/CI/smoke) can live in the repo or under `ecosystem/` or be generated by a rake task.
- The reconciliation logic + the "Winston standard" definition are what travel with the code.

## Consumer (WUT) Side
- WUT will gain a Parquet/DM adapter (reusing its DatasetLoader bulk patterns and indicator logic where helpful).
- It can load via HTTP from DM or (preferred for perf in podman dev) by reading the mounted parquet file directly (DuckDB gem shines here).
- WUT keeps its local `activities` + existing Yahoo/CSV paths during transition; the DM parquet path becomes the preferred "clean 3y+ with derivatives" source for daily ops and new work.
